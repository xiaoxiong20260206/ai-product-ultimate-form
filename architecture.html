⚡ 四类技能体系详解（hover 看细节）

Ⅰ 原生 Skill (Native)

判定标准：主要靠文本/结构化推理即可完成；不依赖外部API；无需并行拆分

执行方式：主Agent读取SKILL.md后直接执行（可被自然语言“教会”）

失败模式：信息源不足 / 指令不够具体 / 需要外部动作却没工具

📊 行业调研 📝 文档撰写 📑 PPT制作 🔍 深度分析

Ⅱ 工具 Skill (Tool/MCP)

判定标准：需要调用外部服务一次或少量次即可完成（生成/查询/执行）

执行方式：主Agent→MCP网关→外部API；网关统一鉴权/限流/路由

失败模式：鉴权失败 / 限流超时 / 外部服务波动

🖼️ 图片生成 🎬 视频生成 🔄 CI/CD 📈 数据查询

Ⅲ 任务 Skill (Task)

判定标准：可拆成多个相互独立的子任务并行（多视角/多维度/多分片）

执行方式：分身引擎派生N个实例；共享Memory快照、独立上下文；结果汇总+经验回流

失败模式：拆分不合理 / 汇总冲突 / 并发资源不足

🔎 CR代码审查 ♻️ 重构评审 📐 多维需求分析

Ⅳ 流程 Skill (Flow)

判定标准：存在多阶段依赖链（Stage DAG），且需要过程可见、可中断、可重试

执行方式：两层编排：Flow层(阶段) + Task层(阶段内执行任意Skill)

失败模式：阶段依赖不清 / 数据传递缺失 / 断点恢复不完整

🚀 需求→发布全链路 🏗️ 项目初始化 📦 批量迁移
🧭 Skill 类型决策指南（P1）
快速判断
只要写/想？
→ 先用 Native
要调API？
→ 用 Tool（走MCP网关）
可并行拆？
→ 用 Task（分身并行）
有阶段依赖？
→ 用 Flow（阶段编排+断点）
建议产品侧在PRD里写清：触发词、输入约束、输出形态、失败兜底；技术侧写清：依赖、权限、SLA、回滚。
🧬 Skill 生命周期与治理（P1/P2）
开发 → 注册(元数据/版本) → 灰度/AB → 评估(Eval) → 发布 → 回滚/下线
关键治理项：Schema校验、权限审计、成本预算(Token/调用)、SLA、观测面板、风险开关。
🧠 Memory 分层与隔离策略（P0）
L1 工作记忆
会话上下文/临时变量
特点：低延迟；自动过期；不做长期沉淀
L2 情景记忆
偏好/历史/任务记录
特点：可审计；可删除；支持时间衰减
L3 语义记忆
知识/文档/经验索引
特点：语义检索；去重聚合；引用溯源
隔离：租户（企业/个人）→ 产品（IDE/Web/助手）→ 用户；共享仅限“通用偏好/匿名经验”，敏感信息必须可删除、可审计。
